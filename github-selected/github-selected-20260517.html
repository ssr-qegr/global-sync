🚀 项目一：Cua - 让 AI 真正"会用电脑"的开源基础设施

项目基本信息

  • 项目名称：trycua/cua
  • GitHub Stars：14.8k+（持续快速增长）
  • YC 支持：Y Combinator 投资项目
  • 主要功能：开源的 Computer-Use Agent 基础设施，提供沙盒环境、SDK 工具包和评测基准，让 AI Agent 能够安全、高效地控制完整的桌面系统（macOS/Linux/Windows/Android）
  • 项目地址：https://github.com/trycua/cua

技术栈分析

组件 技术 用途
虚拟化层 Apple Virtualization.framework + QEMU 高性能 macOS/Linux 虚拟机（97% 原生速度）
容器层 Docker/KVM 轻量级 Linux 沙盒环境
SDK 层 Python/TypeScript 统一 API 封装鼠标/键盘/截图/Shell/文件 I/O
Agent 层 LangChain 风格 ORA 循环 Observe-Reason-Act 自主决策执行
评测层 Cua-Bench OSWorld/ScreenSpot 等基准测试支持
协议层 MCP Server 与 Claude Desktop/Cursor 等工具集成

核心架构亮点：

✅ 三层解耦设计：AI 模型无需关心底层操作系统差异，通过统一 API 操作所有环境
✅ 沙盒即服务：云沙盒热启动 < 1 秒，支持快照和并行 fork
✅ 多运行时支持：同一 API 适配 5 种操作系统 + 自定义镜像

落地价值说明

为什么这是 AI Agent 工程化的里程碑项目？

  1. 填补基础设施空白：当 AI 模型具备了"看屏幕、点鼠标、敲键盘"的能力后，Cua 提供了第一个生产级的安全操作环境。不再需要给 Agent 直接访问宿主系统的权限，所有操作都在隔离沙箱中完成。

  2. 从 Demo 到 Production 的关键一步：传统的 browser-use/playwright 方案只能控制浏览器，而 Cua 实现了完整桌面环境的自动化，支持：
    - Xcode/iOS 开发自动化
    - Windows 企业应用测试
    - Android App 自动化
    - 跨平台工作流编排

  3. 训练数据飞轮：内置的 Cua-Bench 不仅能评测 Agent 性能，还能记录完整的操作轨迹，直接导出为强化学习训练数据，形成"评测→训练→优化"的闭环。

  4. 企业级安全设计：
    - 完全隔离的沙箱环境
    - 操作全程可审计回放
    - 资源配额细粒度控制
    - 支持私有部署

真实使用场景：
- 🎮 游戏 AI：在 Windows 沙箱中训练游戏操作 Agent
- 🧪 QA 自动化：跨平台 App 自动化测试（iOS/Android/macOS/Windows）
- 💼 企业流程自动化：ERP/CRM 等桌面应用的自动化操作
- 📚 AI 培训：标准化的 Agent 训练环境，确保可复现

可以学习到的核心知识点

  3. 统一 API 设计哲学：如何用一套接口适配 5 种不同操作系统的底层差异（策略模式 + 适配器模式）
  4. 评测基准设计：从任务定义、执行、评判到数据导出的完整评测体系设计

快速上手

# 方式一：Python SDK（推荐开发者）
# 需要 Python 3.11+
pip install cua

# 第一个沙盒 Agent
from cua import Sandbox, Image

async with Sandbox.ephemeral(Image.linux()) as sb:
    # 执行 Shell 命令
    result = await sb.shell.run("echo hello")
    # 截图
    screenshot = await sb.screenshot()
    # 鼠标点击
    await sb.mouse.click(100, 200)
    # 键盘输入
    await sb.keyboard.type("Hello from Cua!")

# 方式二：cuabot（快速体验）
npm install -g cuabot

# 在沙盒中运行 Claude Code
cuabot claude

# 在沙盒中打开 Chromium
cuabot chromium

# 直接操控沙盒
cuabot --screenshot          # 截图
cuabot --type "hello world"  # 输入文字

# 方式三：Lume（macOS 本地高性能 VM）
# 一键安装 Lume CLI
/bin/bash -c "$(curl -fsSL https://raw.githubusercontent.com/trycua/cua/main/libs/lume/scripts/install.sh)"

# 拉取并启动 macOS Sequoia 虚拟机
lume run macos-sequoia-vanilla:latest

☸️ 项目二：Kubernetes 1.36 - AI 工作负载原生支持里程碑

项目基本信息

  • 版本代号：Haru（2026 年首个重要版本）
  • 发布时间：2026 年 5 月 13 日
  • 核心价值：Kubernetes 首次原生支持分布式 AI 训练工作负载的编排，标志着云原生从"容器编排"向"AI 基础设施"的演进
  • 官方博客：Kubernetes v1.36: Advancing Workload-Aware Scheduling

技术栈分析

AI 工作负载相关核心更新：

功能 阶段 解决的痛点
Workload-Aware Preemption Alpha 分布式训练"部分抢占"死锁问题：以往高优先级任务抢占单个 Pod，导致 8 卡训练任务 7 卡在运行但无法推进
PodGroup Gang Scheduling Beta 原子化调度：将整个训练任务的 Pod 作为一个整体，要么全部调度成功，要么全部等待
DRA for PodGroups Beta GPU 设备共享：PodGroup 内的所有 Pod 共享同一张 ResourceClaim，突破单 Pod 256 设备限制
Job Controller Integration Alpha 开箱即用：Job 控制器自动创建 Workload 和 PodGroup，无需额外安装 Volcano/KubeBatch
In-Place Vertical Scaling Beta 资源动态调整：不重启容器即可调整 CPU/内存/GPU 配额
Memory QoS via cgroup v2 Beta 分层内存保护：更好地隔离训练任务和系统进程的内存争用

对 AI 基础设施的意义：

问题 v1.35 及之前 v1.36 之后
分布式训练调度 需要安装第三方调度器（Volcano/KubeBatch） 原生支持，开箱即用
GPU 资源分配 整卡分配，资源浪费严重 DRA 分区/共享，支持 MIG vGPU
任务抢占 单 Pod 抢占，容易死锁 PodGroup 整体抢占，原子性操作
资源调整 必须重启 Pod 才能调整资源 原地垂直扩缩容，不中断训练

落地价值说明

为什么 Kubernetes 1.36 是 AI 基础设施工程师的必修课？

  1. 告别调度器碎片化：过去两年 AI 团队不得不在 Kubernetes 之上叠加 Volcano、KubeBatch、Yunikorn 等第三方调度器，导致运维复杂度飙升。1.36 将 Gang Scheduling 原生化，AI 工作负载终于可以"用标准 Kubernetes"运行。

  2. 解决分布式训练的核心痛点：
    - 死锁问题：工作负载感知抢占确保高优先级任务要么完整获得所有资源，要么不抢占任何资源，彻底解决"7/8 卡运行但无法推进"的经典死锁
    - 规模问题：PodGroup 级别的 ResourceClaim 支持数千个 GPU 的统一管理，突破了之前单 Pod 256 设备的限制
    - 弹性问题：原地垂直扩缩容让训练任务可以在不中断的情况下调整 GPU 配额，配合 Spot 实例大幅降低成本

  3. 从"容器编排"到"AI 基础设施"的范式转移：
    传统 Kubernetes 设计面向无状态 Web 服务和有状态数据库，Pod 之间是相互独立的。而 AI 训练任务是紧耦合的并行工作负载，Pod 之间有强依赖关系。1.36 引入的 Workload/PodGroup 分离架构，首次在原生层面承认了这种差异。

  4. 生态统一的信号：
    NVIDIA、AMD、Intel 等硬件厂商都在基于 DRA（动态资源分配）开发标准设备插件。1.36 的这些更新意味着 AI 基础设施终于有了标准化的上游接口，不再需要厂商专属的调度器扩展。

可以学习到的核心知识点

  1. Gang Scheduling 原理：成组调度的设计思想，如何在分布式系统中实现原子性资源分配
  2. DRA 设备插件开发：动态资源分配的编程模型，如何为自定义硬件开发 Kubernetes 设备插件
  3. 调度器扩展点设计：Preemption、Score、Filter 等调度器扩展点的最佳实践
  4. AI 工作负载编排模式：分布式训练、参数服务器、All-Reduce 等架构在 Kubernetes 上的最佳实践
  5. 资源隔离与 QoS：cgroup v2 的内存 QoS 机制，如何在多租户集群中保障训练任务稳定性

适合人群

  • ✅ 云原生/平台工程师（构建企业级 AI 基础设施）
  • ✅ MLOps 工程师（优化大规模训练任务的调度效率）
  • ✅ Kubernetes 贡献者（了解上游调度器的演进方向）
  • ✅ AI 基础设施架构师（设计标准化的 AI 训练平台）

快速体验新功能

# 启用 WorkloadWithJob 特性门控后，这样的 Job 会自动获得 Gang Scheduling
apiVersion: batch/v1
kind: Job
metadata:
  name: distributed-training
spec:
  completionMode: Indexed  # 必须：每个 Pod 有稳定身份
  parallelism: 8           # 8 卡并行训练
  completions: 8           # 必须：parallelism == completions
  template:
    spec:
      restartPolicy: OnFailure
      containers:
      - name: trainer
        image: pytorch/pytorch:2.5-cuda12.4
        resources:
          limits:
            nvidia.com/gpu: 1  # 每个 Pod 1 张 GPU

1.36 会自动为这个 Job 创建：
- 一个 Workload 对象（静态模板）
- 一个 PodGroup 对象（运行时状态）
- 自动设置 schedulingGroup 字段
- kube-scheduler 将 8 个 Pod 作为一个整体调度

# 查看 PodGroup 状态
kubectl get podgroups

# 查看调度事件（可以看到成组调度的日志）
kubectl describe podgroup distributed-training-pg
